# Agent Skill 评估

## 结论

当前这套设计已经不只是“可路由”，而且已经开始具备“有界加载”的专业 Agent Skill 形态。

它的强项已经很完整：有明确路由键，有知识分层，有场景因子化，有反教条输出，有对照式 reference pack，也有正式的上下文加载策略和仓库内 fallback tier 指引。当前剩下的风险，更多是长期治理风险，而不是基础设计缺口。

## 当前做得好的地方

- `intent x medium x stage x output x constraints` 这个路由键是真能用的，不是空口号。
- `knowledge_atom` / `workflow_protocol` / `evaluation_rubric` 三层分离，让理论、流程和评估不会混成一坨。
- `scenario taxonomy` 和场景图谱把“剧本问题到底是什么”先拆开了，避免一上来就落入题材口号。
- `path_options`、`boundary_map`、`scope_correction`、`pattern_reference_pack` 这几个输出，把发散、边界、范围修正和范式参考都明确了。
- `context_loading_plan` 和 loading modes 让“到底该加载多少”第一次变成了可执行机制。
- `voice_style_guide` 和表达镜头触发规则，让”角色声纹 / IP 一致性 / 活人感”第一次变成了可明确调用的能力，而不是散落在对白建议里的附带话术。
- `team_workflow_blueprint` 和 team-mode matrix 让“应该怎么组织多智能体 / writers' room / 审查链”第一次变成正式能力，而不是只靠临时发挥。
- 社区反馈回路和苏格拉底式问题总表，让“质疑”和“反例”变成系统输入，而不是情绪噪音。
- 仓库内的 [`docs/shared/agent-tiers.md`](./shared/agent-tiers.md) 已经补上，tier 规则不再依赖缺失文件。
- route overlap 检查已经进 CI，这对防止相邻输出慢慢糊成一团很关键。

## 主要脆弱点

### 1. 加载治理现在有了，但还可能被架空

仓库现在已经有 loading ladder，但真正的风险是：以后大家继续加资产时，嘴上承认 bounded loading，实际却默认把更宽的包当常规做法。

如果发生这种漂移，坏结果还是那两个：
- 加载太少，永远停在 `route_kernel` / `focus_pack`，回答又会重新变泛；
- 加载太多，动不动就升到 `teaching_pack` / `survey_pack`，上下文还是会被腐化。

### 2. 路由容易互相靠近

以下几个输出都很有用，但概念距离很近：
- `path_options`
- `boundary_map`
- `scope_correction`
- `pattern_reference_pack`
- `team_workflow_blueprint`

如果路由治理不够严格，模型还是会把它们都当成”更聪明一点的回答模式”。现在 overlap check 降低了静默歧义，但不能代替维护者判断。

### 3. 参考材料容易过载

现在仓库里已经有：
- 场景图谱
- 现实镜头
- 范式参考包
- 案例卡
- 反教条文档

这些材料都很有价值，但不是每个任务都应该全量加载。没有加载治理的话，Agent 会倾向于“多拿一点总没错”，最后变成上下文过多、冗余太重、注意力被稀释。

### 4. 场景轴有长胖风险

场景图谱现在已经很好用了，但也因此更要防止 axis creep。场景卡可以持续追加，轴不能轻易无限增加，否则场景系统会慢慢变成另一套隐性本体教条。

## 针对性优化建议

1. 让上下文加载模式长期保持“真有效”。
   - `route_kernel`
   - `focus_pack`
   - `compare_pack`
   - `teaching_pack`
   - `survey_pack`
   模式已经有了，下一步是防止新功能绕过它们。

2. 给路由加优先级和冲突测试。
   - 如果两个 route 会同时命中，必须规定谁优先。
   - 如果新增 route 让旧路由变得含糊，应该先让测试失败，而不是让用户承担歧义。

3. 冻结场景 taxonomy 的核心轴。
   - 场景卡可以继续加，应该是追加式。
   - 新增 axis 需要审查，因为 axis 一旦无限膨胀，taxonomy 就会变成第二套“隐性教条”。

4. 给 reference pack 加入口控制。
   - 只在教学、对比、诊断、解释“为什么这个更好”的场景下加载。
   - 普通写作任务不要默认带上参考包。

5. 加一个加载预算规则。
   - 如果新增材料不会改变路由、评分或下一步决策，就不要加载。
   - 如果 Agent 说不清楚“为什么这份参考会改变答案”，这份参考大概率就是冗余的。

6. 把 tier fallback 控制在小而够用的规模。
   - fallback 已经补上。
   - 但不要再把它长成第二套复杂流程框架。

7. 表达层要保持克制。
   - 风格资产应该在真正改变措辞决策时加载。
   - 不要把“需要活人感”误用成给所有请求加一层风格资料。

8. team mode 也要保持克制。
   - 只有当协作结构真的会改变下一步判断时，才启用 team blueprint。
   - 不要把所有普通写作请求都包装成多智能体表演。

## 最合适的平衡点

最好的平衡不是“尽可能多加载”，也不是“永远只加载最少”。

更好的规则是：
- 先加载能改变路由的最小集合；
- 只有在对比、诊断、教学这三类任务里才扩展；
- 一旦再加一个资产也不会明显改变答案，就停止加载。

这才是“既不腐化上下文，又能找到足够多相关参考”的实际平衡点。

## 为什么这套设计还有很强的扩展性

这套架构的可扩展性其实很好，因为它是追加式的：
- 新媒介可以加；
- 新类型卡可以加；
- 新范式参考包可以加；
- 新 protocol 和 rubric 也可以加，只要还是可路由的。

真正要控制的不是“能不能加”，而是“加了以后有没有提升辨识力”。

## 最优先的六件事

- 让上下文加载治理长期保持有效。
- 保持仓库内 tier fallback 简洁可用。
- 持续维护 route 冲突检测。
- 保持场景 taxonomy 的轴稳定，不要让它无限长胖。
- 把 voice guide 当成校准支架，而不是替代成稿的统一腔调。
- 把 team blueprint 当成工作流选择器，而不是给所有请求套的总包装。
