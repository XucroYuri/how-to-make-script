# 私有参考蒸馏政策

仓库在编写阶段可以从私有工作笔记里学习，但仓库发布后不能继续依赖这些私有材料才能成立。

## 基本原则

私有参考只能作为临时蒸馏输入，不能变成公开仓库的隐藏依赖、默认前提或不可访问引用。

## 允许做的事

- 从私有笔记中抽取可复用的方法机制；
- 把这些机制改写成公开的 `knowledge_atom`、`workflow_protocol`、`evaluation_rubric`、`example_fixture`；
- 用合成示例替换私有案例或工作模板；
- 当某类方法存在合适的公开参考时，补充公网 source map。

## 不允许做的事

- 在公开仓库里留下仓库外的本地绝对路径链接；
- 把私有笔记伪装成公开权威来源；
- 不做抽象就直接搬入私有 prompt 壳、工作模板或本地流程；
- 让未来的协作者必须访问私有笔记，才能理解仓库里的公开资产。

## 转译标准

只要私有笔记影响了仓库，公开版本至少要做到这几件事：

1. 说明可复用机制，而不是照抄本地文档标题；
2. 把长期成立的规则和临时示例分开；
3. 删掉工具绑定、个人绑定、本地环境绑定的细节；
4. 在更诚实的情况下，把 `source_status` 标成 `derived` 或 `synthesized`；
5. 写清作用边界、竞争路径或常见误用，避免笔记被固化成假普遍规律。

## 公网来源优先

如果某类方法有合适的公开参考，应记录到 source map 里。
这些链接是“帮助定位的入口”，不是“唯一正确的证明”。

## 需要持续避免的失败模式

最危险的情况是形成”秘密单一来源”：

- 仓库表面上看起来完整；
- 真正的方法逻辑仍藏在私有库里；
- 未来协作者既无法审计，也无法诚实扩展。

这个仓库应始终优先选择可公开审查的抽象资产，而不是隐藏权威。
