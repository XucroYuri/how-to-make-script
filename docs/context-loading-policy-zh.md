# 上下文加载策略

这份策略放在“路由之后、生成之前”。它不负责选路由，只负责在路由已经选定后，控制到底加载多少上下文，避免把模型喂成“会复述仓库”的状态。

## 评估

当前项目在 Agent Skill 设计上有几个明显优点：
- 路由先行，先定 intent / medium / stage / output / constraints；
- 场景图谱、参考范式、边界工具、范围修正工具已经分层；
- content model 倾向于“小资产”，而不是一口气塞一个大 prompt。

真正的风险不是知识不够，而是加载过量。只要请求一复杂就把附近所有文档都拉进来，模型就会开始复述仓库，而不是使用仓库。这会带来三类问题：
- 上下文腐化，关键信号被噪音淹没；
- 推理变慢，创意推进被冗余材料拖住；
- 输出变钝，答案看起来“很全”，但不再“很准”。

所以这套策略的目标，是在准确性和广度之间保住一个可控平衡。

## 基本原则

1. 先路由，再加载。
2. 先加载最小能改变判断的上下文包。
3. 一次只扩一层，不要从一个 route 直接跳进整个知识仓。
4. 优先加载“对照型参考”，而不是批量加载相似材料。
5. 一旦新上下文不再改变路由、输出契约或自检，就停止扩展。

## 加载模式

具体模式放在 [`references/context-loading-modes.md`](../references/context-loading-modes.md)。

简化来说：
- `route_kernel` 是最小可路由包；
- `focus_pack` 是单任务、单路由、单 rubric 的执行包；
- `compare_pack` 用于备选方案、为什么不这样写、边界判断；
- `teaching_pack` 用于解释、示例和学习；
- `survey_pack` 只在用户明确要“全景式梳理”时才用。

## 扩展触发

只有在“新增一层上下文真的会改变答案”时才扩展：
- 任务是混合型，单一路由不够；
- 用户明确要 alternatives、counterexample、为什么更好；
- 草稿已经暴露出路由错配、失败层或边界混淆；
- 任务涉及受众适配、委制适配、学习路径、现实约束等现实维度；
- 任务明确需要风格校准、语域控制、IP 声纹延续或更强的活人感；
- 任务明确需要 team mode 选择、handoff 设计或 review gate 设计；
- 用户明确要求教学、比较或广泛综述。

## 停止条件

满足任一条件就停：
- 路由已经稳定，输出契约已经固定；
- 新材料只是重复已有结论；
- 再加载一层只会增加细节，不会改变判断；
- 当前上下文包已经足够包含“路由锚点 + 支撑参考 + 反例/对照”。

如果任务很窄，就在第一包里停，不要硬凑“更全面”。如果任务很宽，就停在“能讲清 tradeoff 的最小包”。

## 失效信号

以下信号说明加载过量或过少：
- 模型开始复述文档，而不是解决问题；
- 多个来源彼此冲突，但输出没有把冲突消解掉；
- 由于 breadth 太大，输出变得泛；
- 由于只看一个样本没有对照，输出变脆；
- 模型开始把参考片段当模板，而不是当参考路径。

## 平衡规则

- 始终保留一个主 route pack。
- 除非用户明确要更广的 survey，否则最多再加一个相邻比较包。
- 写作类问题优先采用“一个强例子 + 一个弱对照”，不要用很多相似例子堆量。
- 处理边界问题时，优先用 `boundary_map` 或 `scope_correction`，不要先去加载一大堆通用 craft 文档。
- 处理场景问题时，优先看 [`docs/scenario-atlas-zh.md`](../docs/scenario-atlas-zh.md) 和 [`references/scenario-taxonomy.json`](../references/scenario-taxonomy.json)，再决定要不要继续加样本。
- 处理受众 / 策略问题时，先加载现实镜头，再考虑扩展到更广的 craft。
- 处理角色声纹 / 语言风格问题时，先加载最小表达镜头，不要直接上大包参考样本。
- 处理协作机制问题时，先加载最小 team-mode bundle，不要直接把多智能体框架比较全塞进来。

## 回退规则

如果加载太宽：
- 退回 route kernel；
- 只重载主 protocol、主 rubric 和一个支撑参考；
- 如果 route 还是不清楚，就直接问一个最小澄清问题。

如果加载太窄：
- 补回缺失的决策层；
- 再加一个对照参考或边界说明；
- 不要用“加载整个仓库”来修补 under-loading。

## 一句话

最好的上下文包，是能让模型把路由说清、把判断讲明、把答案写对的那个最小包。
