# 认识论姿态

这个仓库不应该表现得像“剧本创作教义手册”。

它要做的，不是证明世界上只有一种正确写法，而是把创作判断变得更可解释、更可质疑、更可复用，同时承认剧本创作本来就建立在复杂社会、复杂媒介、复杂时代和复杂个体差异之上。

## 核心立场

这里的剧本知识应该被理解为：

- 启发式，不是绝对真理；
- 有边界，不是上下文无关；
- 有时代性，不是永恒规则；
- 会被媒介塑形，不是跨媒介完全通用；
- 可以被挑战，不是因为写进仓库就自动成立。

也就是说，一个 atom、protocol 或 rubric 可以很有用，但依然必须允许反例和修正存在。

## 这会改变什么

对维护者和 Agent 来说，默认应该这样做：

- 在约束还没有把路径压缩到只剩一种时，不要过早收敛成单解；
- 先分清什么是硬边界，什么只是软约束；
- 即使主规则不改，也保留有价值的反例；
- 如果一条规则还有存活核心，优先做范围修正，而不是简单翻成反向口号；
- 随时追问：这条规则到底只适用于某个媒介、某个平台、某一类观众、某个时代，还是更普遍？
- 在 `discover / design` 阶段允许多条有效路径并存。

## 哪些表述值得警惕

下面这些说法，一出现就应该自动触发质疑：

- “一定要”
- “绝对不能”
- “真正的编剧都……”
- “观众永远会……”
- “这个类型只能这样写”
- “只有这一种结构才成立”

这些句子有时不是完全错，但它们几乎总是缺少适用边界。

## 仓库更希望怎么做

当问题还没有被约束压到单一路径时：

- 返回 2-4 条可行路径，而不是假装只有一条；
- 讲清楚这些路径之间真正的 tradeoff；
- 讲清楚什么约束会淘汰哪条路径；
- 标明哪些判断是强规则，哪些只是弱经验、局部经验、历史经验。

当一条规则被挑战，但还没坏到必须整体废弃时：

- 先保留最小仍然成立的核心；
- 明确写出它在哪里会失效；
- 把“原来太宽”的规则缩到诚实范围，而不是改写成另一条绝对说法。

当约束已经很强时：

- 就明确收敛；
- 把被排除的路径也说清楚；
- 说明不是“宇宙真理”，而是“在这个上下文里被排除了”。

## 这不意味着什么

反教条不等于：

- 什么都对；
- 什么都不能判断；
- 所有反对意见都一样有价值；
- 仓库应该故意变模糊。

我们不是取消判断，而是拒绝把局部判断冒充成绝对真理。
