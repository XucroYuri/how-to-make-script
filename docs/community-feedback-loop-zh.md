# 社区反馈循环

这个仓库不只是欢迎提 issue。它更想要被挑战。

这里最有价值的社区反馈，不是泛泛的“我觉得不对”，而是能把仓库推向更准确判断的反驳、例证和现场反馈。你可以直接质疑理论、指出路由错误、补充生产现场的反例，或者告诉我们某条规则在真实项目里会怎么坏掉。

仓库里的指导本质上是启发式的、带边界的、可以争论的，而且是受历史条件影响的。也就是说，遇到一个强反驳，不应该把它当噪音，而应该把它当成改进路由、修正 rubric、补足边界图的机会。
很多高质量反驳的正确落点，并不是把旧规则判成“全错”，而是把它缩窄成一个更诚实的版本。

## 我们希望收到什么

有价值的反馈包括：

- 带复现步骤的问题报告；
- 对理论的反驳，最好带反例；
- 来自编剧、策划、剧本医生、导演、制片、开发岗位的现场反馈；
- 路由失败，也就是 `intent x medium x stage x output x constraints` 走错了；
- 现有 taxonomy、workflow、rubric 没覆盖到的死角；
- 专业质疑，说明为什么某条规则太粗、太弱、太乐观；
- 能证明“这条建议在某类项目里会失效”的 counterexample。

## 应该从哪里开始

先选对表面，再开始提：

- 如果还在搞清楚一条说法或 route 到底是什么意思，先去 `Q&A`；
- 如果已经有真实分歧，但还比较开放、还没缩成 issue，先去 `General`；
- 如果你看到的是缺失路径、缺失 workflow 或社区流程问题，先去 `Ideas`；
- 如果你手里有现场案例、项目复盘、真实试错，先去 `Show and tell`；
- 只有当你已经能明确指向具体文件、具体 claim、具体 route、具体 rubric 时，再开 issue。

这样做的目的，是让 issue 区只承接“可以被行动化”的问题，而不是把所有模糊争论都硬塞进去。

## 一条反馈会变成什么

社区输入不应该只停留在评论区，而应该转化成仓库资产：

- 变成新的 `knowledge_atom`，如果缺的是一个判断点；
- 变成新的 `workflow_protocol`，如果缺的是一段决策流程；
- 变成新的 `evaluation_rubric`，如果缺的是清楚的评价标准；
- 变成新的 `example_fixture`，如果缺的是路由稳定性的回归样例；
- 变成文档更新，如果缺的是解释、例子或使用方式；
- 变成治理规则更新，如果缺的是贡献规范、路由规则或安全边界。

除了直接改资产，常见的处理结果还包括：

- 产出一份 `scope_correction`，把一条过宽规则缩窄到真实成立范围；
- 产出一份 `boundary_map`，把“绝对不能碰”和“可后置评审”的内容拆开；
- 产出一组 `path_options`，当问题的本质其实是过早收敛。

## 社区循环

1. 贡献者提交 issue、PR 或讨论，带证据和例子。
2. 维护者先判断这是知识缺口、路由错误、评分缺口、样例缺口，还是治理缺口。
3. 再确认问题是确实存在、表述不清，还是已经在别处覆盖。
4. 把问题收敛成最小可承载的资产类型。
5. 给新资产补链接、补测试、补 fixture，避免同类问题以后再次出现。
6. 只有当仓库已经吸收这条经验后，原 issue 才关闭。

## 运营节奏

- 72 小时内，维护者应该先给线程分类，或把它转到更合适的讨论表面。
- 7 天内，线程应该至少有 `status:*`、补证据请求，或明确下一步转化动作。
- 每周重点看 `needs-triage`、`needs-practitioner-input`、`needs-counterexample`。
- 每月最好做一次短总结，说明哪些反驳真的推动了仓库演化。

## 维护者回应原则

- 反驳默认算贡献，除非它是攻击性的或没有证据。
- 先要证据，再谈措辞。
- 优先问“哪里坏了”，不要先问“你是不是不喜欢”。
- 如果反馈方向是对的，但太大，就把它拆成可落地的资产。
- 如果反馈有价值但信息不够，就保留 issue，并明确缺什么证据。
- 如果 feedback 是一个有价值的反例，即使主规则不改，也要保留它。
- 如果反馈只是削弱了原规则，而没有完全推翻它，优先做范围修正，而不是直接翻成相反口号。

## 我们希望大家这样提

- “这个路由错了，因为输出契约和媒介不匹配。”
- “这条 rubric 漏掉了我在实际开发里经常看到的失败模式。”
- “这条规则适合长片，但放到短剧就不成立。”
- “这个例子太顺了，遮住了真正的权衡点。”
- “这个理论应该拆成两个原子，因为两类判断其实不一样。”
- “你把一条路径写成了唯一答案，但其实还有另一条同样有效。”
- “这里需要先发散，再收敛，不应该过早合并成一个结论。”
- “这个判断只是在某个历史条件下成立，不是永恒真理。”
- “这条规则不是完全错，但范围写得太大了，应该缩窄。”
