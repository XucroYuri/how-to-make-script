# Socratic Question Backlog

This is the repository's intake list for unresolved screenplay knowledge questions.

Use it before expanding `knowledge/`, `skills/`, `rubrics/`, or routing rules. Each item should be answered, split into a new asset, or explicitly deferred with a reason.

## How To Use

- [ ] Which unanswered question would change routing, output contracts, or evaluation rules if answered now?
- [ ] Which gap is specific enough to become a new knowledge atom instead of a prose paragraph?
- [ ] Which missing rule belongs in a protocol, and which belongs in a rubric?
- [ ] Which question should become a fixture because route behavior depends on it?
- [ ] Which unresolved issue should block a release because it affects correctness?

## Ontology And Taxonomy

- [ ] Which screenplay concepts still lack canonical names in the taxonomy?
- [ ] Which concepts are duplicated across files and should be merged or aliased?
- [ ] Which concepts are too large and should be split into smaller atoms?
- [ ] Which media labels are too broad to support precise routing?
- [ ] Which stage labels are too broad to distinguish discovery, design, drafting, and diagnosis?
- [ ] Which constraints should be promoted to first-class route inputs?
- [ ] Which content types should be added as explicit output contracts?

## Foundations

- [ ] What is the smallest useful definition of story goal?
- [ ] What qualifies as conflict pressure instead of a generic obstacle?
- [ ] What makes a chain of events causally strong enough to count as structure?
- [ ] When does a character arc count as actual structural change?
- [ ] How should theme be expressed so it changes choices rather than becoming commentary?
- [ ] Which foundational principles break down by medium or genre?
- [ ] Which foundational failure modes still need dedicated atoms?

## Workflows

- [ ] What is the minimum input contract for each workflow protocol?
- [ ] What should each protocol refuse to do when the request is under-specified?
- [ ] Where should a workflow stop and hand off to the next stage?
- [ ] Which fallback should be attempted first when the primary route fails?
- [ ] Which steps are common enough to abstract into shared guidance?
- [ ] Which steps are medium-specific and should not be generalized?
- [ ] Which transitions between protocols still need explicit handoff rules?

## Edge Cases

## Maintenance And Release

- [ ] Which unanswered questions should block a release because they affect route correctness?
- [ ] Which unanswered questions can safely wait until the next iteration?

## Pluralism, Boundaries, And Premature Convergence

- [ ] Which current rules are really local success patterns that are over-claimed as universals?
- [ ] Which request types should return multiple viable paths instead of one collapsed recommendation?
- [ ] Which questions are better answered with a boundary map than with a binary rule?
- [ ] Which constraints are soft enough to defer during ideation, and which remain hard boundaries even in exploration?
- [ ] Where is the repository still confusing market orthodoxy or prestige taste with actual creative necessity?
- [ ] Which workflows still converge too early and therefore hide productive creative variation?
- [ ] Which fixtures should prove that more than one route can be valid under different constraint sets?

## Epistemic Pluralism

- [ ] Which claims are written like universal truths but should actually remain bounded heuristics?
- [ ] Which routes are being collapsed too early when multiple valid paths still exist?
- [ ] Which assumptions are historical artifacts rather than timeless screenplay laws?
- [ ] Which tradeoffs should be shown explicitly instead of hidden behind a single “best” route?
- [ ] Which false universals most often masquerade as professional common sense?

## Boundary Design

- [ ] Which constraints are hard boundaries that must stay visible in exploration?
- [ ] Which constraints are soft and can be suspended until review?
- [ ] Where exactly should the exploration-to-review handoff happen?
- [ ] Which safety, privacy, non-consent, or abuse boundaries can never be relaxed?
- [ ] Which constraints actually protect output quality rather than suppress creativity?

## Divergence And Convergence

- [ ] Which workflows need an explicit divergence stage before convergence begins?
- [ ] Which protocols close too early and should keep multiple candidate paths alive longer?
- [ ] Which outputs should preserve more than one viable version until final selection?
- [ ] What is the signal that more divergence is no longer adding information?
- [ ] Which stop conditions should force convergence and handoff?

## Historical Contingency

- [ ] Which rules only hold under specific media, market, or industrial conditions?
- [ ] Which classic screenplay patterns should be labeled as historical patterns, not eternal laws?
- [ ] Which industry practices should be downgraded to case notes because the environment changed?
- [ ] How should the repo mark the validity window of a method that only works in a certain era?
- [ ] Which “correct” habits are really yesterday’s local optimum?
