# Exploration vs Review

The repository should separate two different kinds of creative work:

1. expanding the option space;
2. gating, filtering, and validating what survives.

They should not be treated as the same step.
This is a sequencing rule, not a blanket permission slip.

## Exploration Phase

In ideation and early design work, the system should be willing to:

- generate multiple directions;
- test bold tonal, structural, and thematic moves;
- suspend soft conformity pressures such as “would a generic market like this?”, “does this feel respectable?”, or “is this conventional enough?”

This phase exists to widen the search space before convergence.
It does not suspend hard boundaries or legitimize harmful conduct as “just exploration.”

## Review Phase

Later phases reintroduce constraints deliberately:

- audience fit;
- platform fit;
- commissioning and production fit;
- release constraints;
- legal, ethical, and safety review;
- output-contract quality checks.

This phase exists to decide what should survive, not to shrink exploration before it starts.

## Hard Boundaries vs Soft Constraints

The repository should distinguish:

- `hard boundaries`: no-go zones that should not be normalized during exploration either, including direct real-world harm planning, exploitation, non-consensual abuse framing as acceptable conduct, doxxing, or instructions that facilitate serious wrongdoing;
- `soft constraints`: taste norms, market orthodoxy, prestige assumptions, brand comfort, platform conventions, and other pressures that may be temporarily suspended in exploration and reintroduced in review.

If a team or agent cannot say which class a constraint belongs to, it should stop pretending it has a final answer and map the boundary first.

## Operational Rule

If the user is exploring:

- widen first;
- map the boundary conditions;
- defer soft conformity checks until review.

If the user is preparing a publishable or production-facing output:

- run the relevant fit and safety checks;
- collapse the option space;
- make the gating logic explicit.

## Why This Matters

If review constraints dominate too early, the repository becomes timid and repetitive.

If exploration never reconnects to review, the repository becomes unserious and unsafe.

The system should therefore support wide exploration with explicit later-stage gating, not confuse the two.
