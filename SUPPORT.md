# Support

Use the lightest support surface that matches your need.

## Questions and Clarifications

Use GitHub Discussions when you are still trying to understand a claim, route, rubric, or contribution path.

- `Q&A`: clarify a concept, file, or route before opening an issue
- `General`: open disagreement, critique, or interpretation debate
- `Ideas`: missing workflows, rival paths, or repo-process improvements
- `Show and tell`: field notes, case studies, usage reports, or professional evidence

Repository discussions:
- <https://github.com/XucroYuri/how-to-make-script/discussions>

## Issues

Open an issue when you can point to the exact target that should change:
- a file
- a claim
- a route
- a rubric
- a fixture
- a governance rule

Issue forms live in:
- [.github/ISSUE_TEMPLATE](./.github/ISSUE_TEMPLATE)

## Security

Do not open public issues for live security problems.

Use:
- [SECURITY.md](./SECURITY.md)
- <https://github.com/XucroYuri/how-to-make-script/security/advisories/new>

## Community Standards

- [CONTRIBUTING.md](./CONTRIBUTING.md)
- [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md)
- [docs/community-operations.md](./docs/community-operations.md)

## Maintainer Contact

If you are unsure where to start, begin with Discussions.

If you need maintainer attention for something that should not live in a public thread, open a short issue requesting private follow-up and do not include sensitive details.
